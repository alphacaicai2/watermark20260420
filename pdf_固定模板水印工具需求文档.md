# PDF 固定模板水印工具需求文档

## 1. 文档信息

- **文档名称**：PDF 固定模板水印工具需求文档
- **版本**：V1.0 Final
- **定位**：用于指导一个可落地开发的 PDF 文字水印工具实现
- **目标交付**：NPM Package + CLI + MCP Server

---

## 2. 项目背景

当前存在一个高频、重复性的工作流：

针对同一份或同一类 PDF 文件，需要反复添加不同字样的文字水印。水印的视觉样式、位置、透明度、字号、旋转角度等格式保持一致，变化的主要只有水印文案本身。

现阶段该流程如果依赖人工操作，会存在以下问题：

1. 重复劳动多，效率低。
2. 每次手动调整水印时，容易出现格式不一致。
3. 难以接入自动化工作流或 AI Agent。
4. 不便于标准化复用，也不利于后续扩展。

因此，需要建设一个标准化工具，使用户只需输入水印文字，即可自动按固定模板对 PDF 全文档所有页面批量加水印，并支持被不同 Agent 或本地脚本调用。

---

## 3. 项目目标

本项目目标是实现一个围绕“固定模板 PDF 文字水印”场景的小型基础工具，满足以下要求：

1. 用户只需输入水印文字，即可自动生成加好水印的新 PDF。
2. 水印样式默认固定，输出结果稳定一致。
3. 工具可以本地直接使用，也可以被 AI Agent 调用。
4. 交付形式具备可复用性，适合沉淀为长期工作流基础设施。

---

## 4. 产品定位

该产品不是一个通用 PDF 编辑器，也不是一个在线文档协作平台。

其核心定位是：

**一个面向固定模板场景的、可编程调用的 PDF 文字水印工具。**

重点解决的是“高频、重复、标准化”的 PDF 水印生成问题，而不是复杂的 PDF 排版编辑问题。

---

## 5. 目标用户与使用场景

### 5.1 目标用户

- 需要频繁对 PDF 加固定格式水印的个人用户
- 需要把 PDF 处理嵌入自动化流程的运营/投融资/法务/咨询类用户
- 需要通过 Claude、OpenClaude 等 Agent 自动处理文件的高级用户
- 需要把能力沉淀为内部工具的开发者

### 5.2 核心使用场景

#### 场景 A：手动本地使用
用户在终端执行命令，输入 PDF 路径和水印文案，生成带水印的 PDF 文件。

#### 场景 B：固定模板批处理
用户反复对同一份标准 PDF 加不同姓名、机构名、用途说明等水印，例如：
- CONFIDENTIAL
- For Kenji Makiguchi Only
- 仅供内部使用
- Demo Version

#### 场景 C：Agent 自动调用
用户通过 Claude 或 OpenClaude 触发工具，由 Agent 自动完成 PDF 打水印并输出结果文件。

#### 场景 D：嵌入更长流程
该工具作为某个自动化链路中的一个步骤，例如：
- 生成 PDF 后自动加水印
- 对外发送前自动生成带姓名的版本
- 批量为不同对象输出不同水印版本

## 6. 范围定义

### 6.1 V1 必做范围

V1 仅覆盖以下能力：

1. 读取现有 PDF 文件。
2. 接收用户输入的文字水印内容。
3. 按固定模板样式对 PDF 全部页面加文字水印。
4. 输出新的 PDF 文件。
5. 提供 NPM Package 能力供程序调用。
6. 提供 CLI 供本地直接使用。
7. 提供 MCP Server 供 Agent 调用。

### 6.2 V1 不做范围

为保证项目边界清晰，V1 明确不包括以下内容：

1. 图像水印
2. 在线网页前端界面
3. PDF 内容编辑、删页、合并、拆分
4. OCR 识别
5. 智能避让内容排版
6. 不同页面应用不同水印模板
7. 权限管理、多用户系统
8. 云端存储与文件管理系统
9. 批量文件目录扫描和复杂任务编排

---

## 7. 交付形态与总体方案

本项目采用“三层交付”结构：

### 7.1 NPM Package（底层核心）

作为底层能力层，负责真正的 PDF 加水印逻辑。

作用：
- 被 CLI 调用
- 被 MCP Server 调用
- 被其他 Node.js 项目复用

### 7.2 CLI（人工直接调用层）

作为最直接可用的入口，供用户在本地终端中快速执行加水印任务。

作用：
- 提高个人使用效率
- 方便脚本化调用
- 适合本地自动化工作流

### 7.3 MCP Server（Agent 接入层）

作为 AI Agent 工具接入层，供 Claude、OpenClaude 等 Agent 使用。

作用：
- 让 Agent 可通过标准工具调用方式执行 PDF 加水印任务
- 便于纳入更长链路的自动化流程

### 7.4 Skill 的定位

Skill 不作为主交付物，而作为可选说明层存在。

其作用是为特定 Agent 补充使用说明，例如：
- 何时调用该工具
- 默认输入文件路径是什么
- 输出命名规范是什么
- 水印文字如何生成

## 8. 功能需求

## 8.1 核心功能：对 PDF 全页加固定样式文字水印

### 功能描述
用户指定一个已有 PDF 文件和一段水印文字，系统对该 PDF 的每一页应用相同样式的文字水印，最终输出一个新的 PDF 文件。

### 输入
- 输入 PDF 文件路径
- 水印文字
- 可选输出路径
- 可选模板名称

### 输出
- 一个新的、已加水印的 PDF 文件

### 基本规则
1. 默认对所有页面生效。
2. 每页使用相同模板参数。
3. 不修改原文件。
4. 输出新文件。
5. 若未指定输出路径，则自动生成默认输出文件名。

---

## 8.2 水印模板机制

### 功能描述
系统应支持“固定模板”概念，即将水印样式参数固化为可配置模板，默认使用标准模板。

### 模板参数至少包括
- 字号
- 颜色
- 透明度
- 旋转角度
- 相对位置策略
- 页面对齐方式

### V1 设计原则
- 默认只提供一个标准模板 `standard`
- 用户一般无需每次传入样式参数
- 后续可支持多模板扩展，但不影响 V1 简洁性

---

## 8.3 自动输出命名

### 功能描述
当用户未显式指定输出文件路径时，系统应自动生成输出文件名。

### 推荐规则
原文件名 + 水印标识 + 扩展名

例如：
- `source.pdf` → `source__watermarked__CONFIDENTIAL.pdf`
- `deck.pdf` → `deck__watermarked__For-Kenji-Only.pdf`

### 处理要求
- 对不适合文件名的字符进行安全处理
- 避免覆盖原文件
- 同名冲突时可追加时间戳或序号

---

## 8.4 中文与英文字体支持

### 功能描述
工具需支持英文水印和中文水印。

### 要求
1. 默认支持英文字符。
2. 支持通过配置指定自定义字体文件。
3. 当使用中文水印时，系统应优先使用支持中文的字体。
4. 若当前字体不支持目标字符，应给出明确报错，而不是输出乱码文件。

---

## 8.5 页面尺寸自适应

### 功能描述
针对不同尺寸的 PDF 页面，水印位置应按页面宽高动态计算，而非写死绝对坐标。

### 要求
1. 基于每页宽高独立计算位置。
2. 默认以页面中心为锚点。
3. 旋转后的布局应在不同页面尺寸下保持视觉一致。

---

## 8.6 CLI 命令能力

### CLI 目标
为本地使用提供简单、稳定、低学习成本的命令入口。

### 推荐命令示例

```bash
pdfwm add --input ./source.pdf --text "CONFIDENTIAL" --output ./out.pdf
```

或

```bash
pdfwm add --input ./source.pdf --text "仅供内部使用"
```

### 固定模板快捷命令（可选）

```bash
pdfwm standard --input ./source.pdf --text "For Kenji Only"
```

### CLI 参数定义
- `--input`：输入 PDF 路径，必填
- `--text`：水印文字，必填
- `--output`：输出路径，选填
- `--template`：模板名称，选填，默认 `standard`
- `--font`：字体文件路径，选填
- `--config`：配置文件路径，选填

---

## 8.7 NPM Package API 能力

### 目标
提供可被 Node.js 项目直接调用的标准化接口。

### 推荐接口

```ts
watermarkPdf({
  inputPath,
  outputPath,
  text,
  templateName,
  fontPath
})
```

### 入参说明
- `inputPath: string`：输入 PDF 路径
- `outputPath?: string`：输出 PDF 路径
- `text: string`：水印文案
- `templateName?: string`：模板名称
- `fontPath?: string`：字体文件路径

### 返回结果
建议返回：
- 输出路径
- 页面数量
- 使用的模板名
- 执行是否成功

---

## 8.8 MCP Tool 能力

### 目标
为 Claude、OpenClaude 等 Agent 提供可调用工具。

### 工具名称
`add_pdf_watermark`

### 推荐输入 Schema

```json
{
  "type": "object",
  "properties": {
    "input_path": { "type": "string" },
    "watermark_text": { "type": "string" },
    "output_path": { "type": "string" },
    "template": { "type": "string" },
    "font_path": { "type": "string" }
  },
  "required": ["input_path", "watermark_text"]
}
```

### 返回内容建议
- 状态：成功 / 失败
- 输出文件路径
- 页面数量
- 报错信息（如失败）

### Agent 使用原则
MCP 层应尽量保持参数简单，不暴露过多样式细节，以提高 Agent 调用稳定性。

---

## 9. 配置需求

## 9.1 默认配置文件
系统应支持配置文件，例如：`watermark.config.json`

### 推荐内容
- 默认模板名称
- 模板参数
- 默认字体路径
- 默认输出目录
- 文件名规则

### 示例结构

```json
{
  "defaultTemplate": "standard",
  "defaultOutputDir": "./output",
  "templates": {
    "standard": {
      "fontSize": 56,
      "opacity": 0.12,
      "rotation": 45,
      "colorRgb": [0.6, 0.6, 0.6],
      "position": "center"
    }
  }
}
```

## 10. 非功能需求

## 10.1 稳定性
1. 对常规 PDF 文件应稳定处理。
2. 同一输入在相同配置下应输出一致结果。
3. 不能破坏原始 PDF 文件。

## 10.2 易用性
1. 默认参数尽量少。
2. 只输入水印文字即可完成主要工作流。
3. 错误提示要清晰，便于用户或 Agent 修复。

## 10.3 可扩展性
1. 架构上支持后续增加更多模板。
2. 支持未来增加图像水印、批量处理等能力，但不影响当前设计。
3. CLI、NPM、MCP 共用同一套核心逻辑，避免重复实现。

## 10.4 可维护性
1. 采用 TypeScript 实现。
2. 模块拆分清晰。
3. 具备基础单元测试和示例。

## 10.5 跨平台兼容性
至少支持：
- macOS
- Linux
- Windows

## 11. 技术建议

### 建议技术栈
- **语言**：TypeScript
- **运行环境**：Node.js
- **PDF 处理库**：pdf-lib
- **CLI 框架**：commander 或 yargs
- **打包工具**：tsup 或 esbuild
- **MCP 实现**：Node.js 版本 MCP Server

### 架构原则
- 所有实际处理逻辑收敛到 `core`
- CLI 只做参数解析和调用
- MCP 只做 schema 暴露和调用
- 配置与核心逻辑分离

## 13. 异常与报错要求

以下情况必须给出明确错误提示：

1. 输入文件不存在
2. 输入文件不是合法 PDF
3. 输出路径无权限写入
4. 字体文件不存在
5. 字体不支持目标字符
6. 水印文字为空
7. 模板名称不存在

报错信息应满足：
- 可读
- 可定位
- 对 Agent 友好

示例：
- `Input PDF not found: ./files/source.pdf`
- `Watermark text is empty`
- `Font does not support required characters`

## 14. 验收标准

项目验收至少满足以下标准：

### 功能验收
1. 输入一个 PDF 和一段文字，可成功输出带水印的新 PDF。
2. 输出 PDF 的每一页都存在水印。
3. 水印样式符合固定模板要求。
4. 中文水印在指定中文字体下可正常显示。
5. 未指定输出路径时，可自动生成输出文件。

### CLI 验收
1. 通过 CLI 可直接完成完整流程。
2. 参数错误时有明确报错。
3. 帮助命令可正常显示使用说明。

### NPM API 验收
1. 可通过函数调用完成水印生成。
2. 返回值包含必要结果信息。

### MCP 验收
1. Agent 可通过 MCP 工具调用成功执行任务。
2. 返回输出文件路径与状态信息。

### 工程验收
1. 代码结构清晰。
2. README 可指导安装与使用。
3. 至少覆盖核心流程测试。

## 16. 最终结论

本项目最终采用以下方案作为正式需求定义：

**以 TypeScript + Node.js 为技术基础，构建一个“底层 NPM Package + 上层 CLI + 可选 MCP Server”的 PDF 固定模板文字水印工具。**

该工具以“用户只输入水印文案，其余样式保持固定”为核心原则，优先解决高频、重复、标准化的 PDF 水印需求，并确保可以被 Claude、OpenClaude 等 Agent 方便调用。

V1 必须保持边界清晰、能力聚焦、结构简单，先把最核心的固定模板文字水印能力做稳，再为后续扩展留出空间。
